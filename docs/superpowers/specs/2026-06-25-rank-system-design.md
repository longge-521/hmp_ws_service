# 欢乐斗地主 36级排位头衔等级系统设计规范

本文档详述了欢乐斗地主游戏中 36级古风排位头衔与星星系统的设计。

## 1. 业务需求说明

1.  **36级头衔系统**：经典古风官职与身价头衔（包身工、短工...亲王、至尊），用以代表玩家的游戏段位。
2.  **子级别与星星**：除最高级【至尊】外，每个头衔拥有四个子段位 `IV, III, II, I`（用数字 1-4 表示，1 对应 I，4 对应 IV）。提升子段位需赢取并积满对应的“星星”数量（3-5 颗不等）。
3.  **结算逻辑（TDD实现）**：
    *   赢牌加星（有翻倍加双星逻辑），输牌扣星（有低段位新手保护及中段位大段位保护）。
4.  **演示与测试**：支持在页面修改排位信息，以便测试各种升降级和高难度头衔的显示。

## 2. 数据库与数据结构设计

### 2.1 数据库表 `player_profile` 字段扩展
在 `PlayerProfileORM` 中扩展以下三个核心字段：
*   `rank_id = Column(Integer, default=1, nullable=False, comment="段位头衔ID(1-36)")`
    *   1 对应“包身工”，35 对应“亲王”，36 对应“至尊”。
*   `sub_rank = Column(Integer, default=4, nullable=False, comment="子段位(1-4，对应I-IV)")`
*   `stars = Column(Integer, default=0, nullable=False, comment="当前子段位内的累积星星数")`

### 2.2 领域层 `PlayerProfile` 模型扩展
增加相应字段定义，并且添加一个根据 `rank_id` 转换为中文头衔的映射列表。

## 3. 后端结算逻辑与 API 接口设计

### 3.1 36级头衔基础配置
头衔列表常量 `RANK_TITLES` 定义如下：
1.包身工, 2.短工, 3.长工, 4.中农, 5.富农, 6.掌柜, 7.商人, 8.小财主, 9.大财主, 10.县尉, 11.县丞, 12.县令, 13.通判, 14.主事, 15.知府, 16.员外郎, 17.郎中, 18.侍郎, 19.巡抚, 20.总督, 21.尚书, 22.大学士, 23.太保, 24.太傅, 25.太师, 26.三等伯, 27.二等伯, 28.一等伯, 29.三等侯, 30.二等侯, 31.一等侯, 32.辅国公, 33.镇国公, 34.郡王, 35.亲王, 36.至尊

### 3.2 结算升降星状态机 (`SQLGameRepository.update_rank_stats`)
根据游戏胜负执行状态变化：
1.  **取各级满星数限制**：
    *   1 - 9 级（包身工 - 大财主）：小段位满星为 **3**，输牌不扣星，不掉段。
    *   10 - 21 级（县尉 - 尚书）：小段位满星为 **4**，输牌扣 1 星，大段位有保护（即不会退回前一个大头衔）。
    *   22 - 35 级（大学士 - 亲王）：小段位满星为 **5**，输牌扣 1 星，无保护（允许掉出当前大头衔）。
    *   36 级（至尊）：无小段位，输牌不扣星，赢牌星星累加。
2.  **加星逻辑（赢牌）**：
    *   如果本局翻倍倍数较大（例如 `multiplier >= 4`，或使用了炸弹/王炸/春天结算），属于“爆发性胜利”，星星 **+2**；否则普通胜利星星 **+1**。
    *   *进阶计算*：
        *   若 `stars` 溢出满星限制：`stars` 归零或计入溢出部分，同时 `sub_rank` - 1（例如从 IV 变为 III）。
        *   若 `sub_rank` 溢出（小于 1）：`sub_rank` 设为 4，同时 `rank_id` + 1。
        *   若 `rank_id` 超过 35：直接设置为 36（至尊），`sub_rank` 不适用，星星继续增长。
3.  **扣星逻辑（输牌）**：
    *   若处于第 1 - 9 级：不扣星。
    *   若处于 10 - 35 级：星星 -1。
    *   *降级计算*：
        *   若 `stars < 0`：
            *   若 `sub_rank < 4`：`sub_rank` + 1，同时 `stars` 设定为该段位满星数 - 1。
            *   若 `sub_rank == 4`（代表小段位最低）：
                *   若处于 10 - 21 级：触发保护，`stars` 设为 0，不降段。
                *   若处于 22 - 35 级：`rank_id` - 1，`sub_rank` 设为 1，`stars` 设定为前一个段位大级别的满星数 - 1。

### 3.3 修改段位 API 路由
*   **接口地址**：`POST /api/game/profile/{player_id}/rank`
*   **请求体**：
    ```json
    {
      "rank_id": 35,
      "sub_rank": 4,
      "stars": 3
    }
    ```
*   在 `SQLGameRepository` 中增加 `update_rank_profile(player_id, rank_id, sub_rank, stars)` 以支持该修改，提供直接的测试体验。

## 4. 前端状态与 UI 渲染

### 4.1 Pinia Store 扩展
*   在 `playerStore.ts` 中增加段位字段：`rankId`、`subRank`、`stars`。
*   添加 `modifyRank(rankId, subRank, stars)` action 方法，用来请求后端修改玩家的排位段位。

### 4.2 大厅界面排位展示
*   在 `LobbyView.vue` 右下角个人信息区，将原静态标签替换为 `RANK_TITLES[rankId] + subRankRoman(subRank)`（例如 `亲王IV`）。
*   将静态星星替换为星条（根据段位上限渲染全部星数，以金星 `★` 和灰星 `☆` 分别渲染当前积累数和缺口数，例如 `★ ★ ★ ☆ ☆ 3/5`）。

### 4.3 个人资料与排行榜同步
*   排行榜弹窗中，也将同步展示每个玩家的头衔名称。

### 4.4 修改段位弹窗
*   在修改欢乐豆的模态框中扩充段位设定选项，让开发和测试人员可以在大厅中一键体验和测试任意段位头衔。
